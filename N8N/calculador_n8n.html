⚠ Observações

Como esta calculadora pensa

Workers são calculados por ⌈concorrência_pico ÷ concorrência_por_worker⌉. A concorrência por worker varia com o perfil — workloads I/O-bound (API leve, misto) suportam 20–25 jobs simultâneos por worker; workloads memory-bound (ETL, IA, Code) ficam em 8–10.

RAM por worker = 512 MB base + (concorrência × mem_por_execução × 1,2 buffer) × margem do usuário. A memória por execução assume workload típico do perfil (n8n é I/O-bound, slot raramente segura payload grande em memória sustentado): 30 MB para API leve, 60 MB para misto, 200 MB para ETL, 250 MB para IA, 150 MB para Code.

Webhook processors são propostos quando o RPS de pico passa de ~5 ou quando webhook concentra mais de 60% do tráfego com RPS > 2 — abaixo disso o main absorve sem problema.

PostgreSQL escala com volume diário (CPU/RAM) e com retenção (storage). O pool de conexões segue workers × concorrência + main + reserva; passar de 100 conexões já merece atenção.

Modo single vs queue é decidido por volume (>1k exec/dia ou >10 concorrentes) ou por requisito de HA. A margem de segurança do usuário é aplicada uma única vez, sobre o resultado já bufferizado internamente — sem dupla margem.

Os números refletem cenário realista de produção, não defensivo. Para produção crítica, instrumente e meça por 2–3 semanas antes de fixar o sizing. Se o seu workload tem outliers de payload (>10 MB), o cálculo escala automaticamente via maxPayload.
