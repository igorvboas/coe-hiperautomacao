# Handoff — Migrations 0014 a 0018 (apply manual)

**Modo:** write-only (apply manual no Supabase Cloud SQL Editor — NÃO `supabase db push`).
**Pré-requisitos:** 0001..0013 aplicadas.

## Por quê

Pacote de features levantado por comparação com um inventário de referência externo
(COE — COPA ENERGIA): RBAC somente-leitura por usuário, 3 status novos no pipeline,
campos operacionais (Azure Boards, linguagem, criticidade, datas de abertura/fechamento
no COE), e 3 tabelas novas (documentos anexados, anotações estruturadas, histórico de
alterações). Ver `.planning/PROJECT.md` para o contexto completo da decisão.

## ⚠️ Ordem de aplicação é OBRIGATÓRIA e cada arquivo roda em um Run SEPARADO

O Postgres não deixa usar um valor de enum recém-adicionado (`ALTER TYPE ... ADD VALUE`)
na mesma transação em que ele foi criado. Por isso 0014/0016 (que só adicionam valores de
enum) são arquivos isolados — **cole e rode cada um sozinho, espere terminar, DEPOIS cole
o próximo.** Não colar dois arquivos juntos no mesmo Run.

| # | Arquivo | O que faz |
|---|---|---|
| 1 | `0014_rbac_viewer_enum.sql` | Adiciona `'viewer'` ao enum `tenant_role`. |
| 2 | `0015_rbac_viewer_policies.sql` | `current_user_role()` + policies de insert/update/delete em `opportunities`/`opportunity_phases`/`opportunity_risks` passam a exigir `role <> 'viewer'`. |
| 3 | `0016_status_enum_v03.sql` | Adiciona `'gestao'`, `'manutencao'`, `'descontinuado'` ao enum `opportunity_status` (8 → 11). |
| 4 | `0017_schema_evolution_v03.sql` | Enum `criticidade_level` + coluna `criticidade`; colunas `azure_boards_codigo`, `linguagem`, `execucao`, `usuarios_servico`, `execucoes_mes`, `data_conclusao`, `data_abertura_coe`, `data_fechamento_coe`; trigger de abertura/fechamento automático no COE; **corrige** `sync_opportunity_phase()` (0004) para não quebrar com os 3 status novos de 0016. |
| 5 | `0018_documentos_anotacoes_historico.sql` | Tabelas `opportunity_documents`, `opportunity_notes`, `opportunity_history` (tenant_id + RLS); bucket privado `opportunity-documents` no Storage + policies por pasta = tenant_id. |

## Como aplicar

1. Supabase Dashboard → projeto → **SQL Editor**.
2. Para cada arquivo, na ordem da tabela acima: colar o conteúdo **inteiro** do arquivo e **Run**. Confirmar sem erro antes de colar o próximo.
3. Depois dos 5, rodar as queries de verificação abaixo.

## Verificação pós-apply

```sql
-- 1. tenant_role tem 3 valores
select enum_range(null::tenant_role);
-- esperado: {member,tenant_admin,viewer}

-- 2. opportunity_status tem 11 valores
select enum_range(null::opportunity_status);
-- esperado: {novo,em_analise,planejamento,backlog,desenvolvimento,homologacao,producao,concluido,gestao,manutencao,descontinuado}

-- 3. colunas novas existem
select column_name from information_schema.columns
where table_name = 'opportunities'
  and column_name in ('criticidade','azure_boards_codigo','linguagem','execucao',
    'usuarios_servico','execucoes_mes','data_conclusao','data_abertura_coe','data_fechamento_coe')
order by column_name;
-- esperado: 9 linhas

-- 4. abertura automática funciona (INSERT de teste)
insert into opportunities (tenant_id, source, solicitante, area, processo, esforco, complexidade, tempo, objetivo)
select id, 'formulario', 'smoke-handoff', 'TI', 'smoke handoff 0014-0018', 'medio', 'medio', 'mensal', 3
from tenants limit 1
returning data_abertura_coe;
-- esperado: timestamp preenchido (não null)

-- 5. status novo não quebra o trigger de fases (não deve dar erro 22P02)
update opportunities set status = 'gestao' where solicitante = 'smoke-handoff';
update opportunities set status = 'descontinuado' where solicitante = 'smoke-handoff';
select data_fechamento_coe from opportunities where solicitante = 'smoke-handoff';
-- esperado: timestamp preenchido (fechamento automático ao entrar em status terminal)

delete from opportunities where solicitante = 'smoke-handoff'; -- limpa o smoke

-- 6. tabelas novas existem com RLS
select tablename, rowsecurity from pg_tables
where tablename in ('opportunity_documents','opportunity_notes','opportunity_history');
-- esperado: 3 linhas, rowsecurity = true

-- 7. bucket de storage existe
select id, public from storage.buckets where id = 'opportunity-documents';
-- esperado: 1 linha, public = false
```
