# Phase 07.6: Enriquecimento por IA das Oportunidades — Research

**Researched:** 2026-05-26
**Domain:** OpenAI Structured Outputs + Next.js `after()` + Supabase service-role UPDATE assíncrono
**Confidence:** HIGH (key APIs verified against current official docs 2026)

## Summary

A fase 07.6 substitui dois steps de input do usuário (Automação + Priorização) por um **enrichment server-side com OpenAI**, disparado de forma fire-and-forget via `after()` do Next.js 16. As três decisões arquiteturais já estão **locked** em PHASE.md — `after()` + `gpt-4o-mini` + service-role client com WHERE defensivo. Esta pesquisa **não reavalia** essas escolhas; foca em traduzir cada uma em código atualizado contra docs 2026 e em mapear a superfície de refactor.

Quatro descobertas que mudam o plano:

1. **`serviceRoleClient()` NÃO existe em `lib/supabase/server.ts`** — só em `tests/setup/supabase-test-client.ts`. PHASE.md afirma "já existe" — isso é incorreto. O plan precisa criar `serviceRoleClient()` em `lib/supabase/server.ts` com `import 'server-only'` no topo + `auth: { persistSession: false, autoRefreshToken: false }`. Reaproveitar o shape do test client.
2. **`after()` em Server Action**: parâmetros são acessados via closure, não dentro do callback (no caso de Server Action está OK chamar `cookies()`/`headers()` dentro do callback, mas para `tenant_id` derivado da sessão **não** é necessário — o INSERT acabou de retornar a row inteira; passamos `inserted.id + profile.tenant_id` por closure). `after()` é **stable desde v15.1**, importado de `next/server`. Vercel respeita `maxDuration` (default 300s, padrão Fluid Compute) — mais que suficiente para 1 chamada OpenAI (~3-8s).
3. **OpenAI SDK v6.39.0** (latest 2026). Há DUAS APIs viáveis: (a) `client.chat.completions.create({ response_format: { type: 'json_schema', json_schema: { name, schema, strict: true } } })` — JSON literal — ou (b) `client.chat.completions.parse({ response_format: zodResponseFormat(schema, 'name') })` — helper com Zod. **Recomendação: usar (b)** porque o projeto já usa Zod 4.x e o helper devolve `.parsed` tipado + `.refusal` sem boilerplate. Modelo `gpt-4o-mini` **não está deprecated na API** (verificado contra openai.com/api/deprecations 2026).
4. **Migration 0010 segue o padrão de 0008/0009**: `add column if not exists` + CHECK + DROP/CREATE da view `opportunities_with_score` (porque `o.*` na view força DROP quando colunas novas mudam a ordem). Não precisa migrar dados existentes — todos os registros antigos viram `ai_enrichment_status DEFAULT 'pending'` e podem ser re-enriched manualmente ou ficar como `failed` sem impacto.

**Primary recommendation:** Estruturar 5 waves no plan — (Wave 0) infra: `serviceRoleClient()` + npm install openai + migration 0010 → (Wave 1) `lib/ai/prompts.ts` + `lib/ai/enrichment.ts` + Zod schema da resposta → (Wave 2) refactor Server Actions (`createOpportunity` + `createPublicOpportunity`) com `after()` → (Wave 3) refactor wizard interno + form público (remover steps) → (Wave 4) UI do badge + edit no modal → (Wave 5) testes mockados.

## Project Constraints (from CLAUDE.md)

Directivas que o planner DEVE respeitar:

- **Multi-tenant via RLS é existencial** — Toda query toca `tenant_id`. UPDATE pós-IA usa filtro **defensivo triplo**: `WHERE id = $X AND tenant_id = $Y AND ai_enrichment_status = 'pending'` (decisão #3 do PHASE.md).
- **Score NUNCA persistido** — Os 9 campos da IA são *inputs* do `opportunity_score()`. Score continua sendo calculado em runtime na view `opportunities_with_score`. Migration 0010 NÃO toca em `opportunity_score()`.
- **pt-BR para UI; EN para código** — Prompts para OpenAI vão em **inglês** em `lib/ai/prompts.ts` (revisão facilitada; também porque o modelo é mais robusto em EN para extração estruturada). Mensagens de erro pro usuário (`ai_enrichment_status='failed'` UI badge) ficam em **pt-BR**.
- **Tenant_id derivado server-side** — `tenant_id` NUNCA aparece no payload do cliente; vem de `profile.tenant_id` (autenticado) ou de `tenants.slug → id` (formulário público). Defesa Mass Assignment do schema 7.5 fica intacta.
- **Cliente final NÃO sabe que existe IA** — O badge no modal de detalhe é interno (visível só para usuários autenticados do CoE). O formulário público e o wizard interno mostram só "Oportunidade registrada" / "Salvo com sucesso".
- **Não criar painel admin / rotas super-admin** — Modal de detalhe editável aproveita componentes existentes (`AutomacaoStep` / `PriorizacaoStep`) — não criar nova rota.

## User Constraints (from PHASE.md "Decisões locked")

### Locked Decisions

1. **Disparo do enrichment**: `after()` do Next.js (App Router) dentro do Server Action. Roda após a resposta ser enviada ao cliente, na mesma região da função; sem infra extra; falha não bloqueia submit. Trade-off aceito: cold-restart entre INSERT e `after()` pode pular enrichment — mitigação é manter `ai_enrichment_status='pending'` para job de catch-up futuro (backlog 999.x).

2. **Modelo OpenAI**: `gpt-4o-mini` (Chat Completions + `response_format: { type: 'json_schema', json_schema: {...}, strict: true }`). Custo baixo, latência boa, maduro. Sem streaming, sem tools, sem agentes.

3. **Bypass do RLS no UPDATE pós-IA**: client `serviceRoleClient()` com filtro defensivo `WHERE id = $X AND tenant_id = $Y AND ai_enrichment_status = 'pending'` — idempotente, evita race se chamado 2x, e protege contra cross-tenant write mesmo com bug no callsite. Sem SECURITY DEFINER function (mais simples, menos cirurgia no DB; pode migrar para function em phase futura se audit pedir).

### Claude's Discretion

- Sequência de waves (5 sugeridas — ver Implementation Order na seção final).
- Estrutura interna de `lib/ai/prompts.ts` e `lib/ai/enrichment.ts` — desde que `lib/ai/prompts.ts` separe templates para revisão fácil.
- Forma de validar a resposta OpenAI antes de persistir — recomendado: Zod schema dedicado `OpportunityEnrichedFieldsSchema` em `lib/opportunities/schema.ts` ou em `lib/ai/schema.ts`.
- Estratégia exata de mock no Vitest (intercept em `globalThis.fetch` vs mock do módulo `openai`) — ver seção "Vitest Mocking".
- UI exata do badge `ai_enrichment_status` no modal (badge ao lado do título do modal, ou pequeno selo no header — fica a critério visual).
- Que rota chamar pra "forçar re-enrichment" se admin pedir (`/api/opportunities/[id]/enrich` POST? botão no modal?) — **não há requisito** disso no escopo; sugiro não implementar e deixar admin editar manualmente em caso de `failed`.

### Deferred Ideas (OUT OF SCOPE)

- IA visível ao usuário final (botão "gerar com IA") — NÃO.
- Retry automático em falha — 1 tentativa apenas, depois `failed` + admin edita.
- Streaming de resposta — não há UI esperando.
- Fine-tuning / RAG / embeddings — só prompt + structured output.
- IA em outros pontos do produto (sugestão de processos, score automático, classificação de status) — backlog 999.x.
- Painel de observabilidade da IA (taxa de falha, custo, tokens consumidos) — `ai_enrichment_error` chega; dashboard fica pós-MVP.
- Rate limit do OpenAI / token budget per tenant — pós-piloto, se monitoring mostrar abuso.

### Pattern 1: Next.js `after()` em Server Action

**What:** Schedule work to run **após** o response HTTP ser enviado ao cliente. Usa primitiva `waitUntil` da Vercel para estender o lifetime da serverless invocation.

**When to use:** Background tasks que (a) não afetam o response e (b) podem tolerar cold-restart silencioso (cobrir com flag persistida no DB).

**Example (verbatim de docs Next.js 16, adaptado):**
```typescript
// lib/opportunities/actions.ts
'use server';

import { after } from 'next/server';
import { enrichOpportunity } from '@/lib/ai/enrichment';

export async function createOpportunity(input: unknown): Promise<CreateOpportunityResult> {
  // ... Zod parse, auth, INSERT (mesmo padrão atual) ...

  const { data: inserted, error } = await supabase
    .from('opportunities')
    .insert({ /* ... */ })
    .select('id, tenant_id')
    .single();

  if (error || !inserted) {
    return { ok: false, error: `Erro ao criar: ${error?.message ?? 'desconhecido'}` };
  }

  // Capture values via closure ANTES do after() — não dependa de cookies/headers
  // dentro do callback (eles funcionam em Server Action, mas é boa prática
  // capturar fora pra clareza)
  const oppId = inserted.id;
  const tenantId = inserted.tenant_id;

  after(async () => {
    try {
      await enrichOpportunity(oppId, tenantId);
    } catch (e) {
      // after() callback errors são logados pela plataforma (Vercel) mas
      // NÃO crasham o response — defesa adicional: enrichOpportunity()
      // internamente também catch + UPDATE status='failed'
      console.error('[ai/enrichment] after() throw inesperado:', e);
    }
  });

  revalidatePath('/opportunities');
  return { ok: true, id: inserted.id };
}
```
[VERIFIED: nextjs.org/docs/app/api-reference/functions/after (lastUpdated 2026-05-19, version 15.1.0 stable)]

### Pattern 2: OpenAI `chat.completions.parse()` com Zod

**What:** SDK helper que combina structured outputs + Zod schema → retorna `.parsed` tipado direto, com `.refusal` detection embutida.

**When to use:** Sempre que a saída precisa ser JSON tipado validado contra schema; evita boilerplate de `JSON.parse(content)` + Zod-validate manual.

**Example:**
```typescript
// lib/ai/enrichment.ts
import 'server-only';
import OpenAI from 'openai';
import { zodResponseFormat } from 'openai/helpers/zod';
import { z } from 'zod';
import { OpportunityEnrichedFieldsSchema } from './schema';
import { buildEnrichmentPrompt } from './prompts';
import { serviceRoleClient } from '@/lib/supabase/server';

const client = new OpenAI(); // lê OPENAI_API_KEY automaticamente

export async function enrichOpportunity(
  opportunityId: string,
  tenantId: string,
): Promise<void> {
  const sb = serviceRoleClient();

  // 1. Lê row (NÃO incluir tenant_id no prompt — defesa anti-prompt-injection)
  const { data: row, error: readErr } = await sb
    .from('opportunities')
    .select('solicitante, area, subarea, processo, frequencia, volume_medio, tempo_execucao, num_pessoas, formulario_extras, persona_extras, source, request_type')
    .eq('id', opportunityId)
    .eq('tenant_id', tenantId)             // defesa cross-tenant
    .eq('ai_enrichment_status', 'pending') // idempotência: pula se já processou
    .maybeSingle();

  if (readErr || !row) {
    // Não há o que processar — log e sai (não escrever 'failed' aqui,
    // status pode já estar 'enriched' por uma chamada anterior)
    return;
  }

  // 2. Monta prompt (EN, em lib/ai/prompts.ts)
  const { systemPrompt, userPrompt } = buildEnrichmentPrompt(row);

  try {
    const completion = await client.chat.completions.parse({
      model: 'gpt-4o-mini',
      messages: [
        { role: 'system', content: systemPrompt },
        { role: 'user', content: userPrompt },
      ],
      response_format: zodResponseFormat(
        OpportunityEnrichedFieldsSchema,
        'opportunity_enriched_fields',
      ),
    });

    const message = completion.choices[0]?.message;

    if (message?.refusal) {
      await markFailed(sb, opportunityId, tenantId, `refusal: ${message.refusal}`);
      return;
    }

    if (!message?.parsed) {
      await markFailed(sb, opportunityId, tenantId, 'no parsed content');
      return;
    }

    const enriched = message.parsed;

    // 3. UPDATE com WHERE defensivo triplo
    const { error: updateErr } = await sb
      .from('opportunities')
      .update({
        ferramenta: enriched.ferramenta,
        escopo_automacao: enriched.escopo_automacao,
        beneficios_esperados: enriched.beneficios_esperados,
        observacao: enriched.observacao,
        risco: enriched.risco,
        esforco: enriched.esforco,
        complexidade: enriched.complexidade,
        tempo: enriched.tempo,
        objetivo: enriched.objetivo,
        ai_enrichment_status: 'enriched',
        ai_enriched_at: new Date().toISOString(),
        ai_enrichment_error: null,
      })
      .eq('id', opportunityId)
      .eq('tenant_id', tenantId)              // defesa
      .eq('ai_enrichment_status', 'pending'); // idempotência

    if (updateErr) {
      await markFailed(sb, opportunityId, tenantId, updateErr.message);
    }
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'unknown';
    // NUNCA logar tenant_id raw em error — escrubar
    await markFailed(sb, opportunityId, tenantId, msg.substring(0, 500));
  }
}

async function markFailed(
  sb: ReturnType<typeof serviceRoleClient>,
  opportunityId: string,
  tenantId: string,
  errorMsg: string,
): Promise<void> {
  await sb
    .from('opportunities')
    .update({
      ai_enrichment_status: 'failed',
      ai_enrichment_error: errorMsg,
      ai_enriched_at: new Date().toISOString(),
    })
    .eq('id', opportunityId)
    .eq('tenant_id', tenantId)
    .eq('ai_enrichment_status', 'pending');
}
```
[VERIFIED: openai SDK helpers.md no GitHub openai/openai-node — `zodResponseFormat` exportado de `openai/helpers/zod`; `chat.completions.parse()` retorna `ParsedChatCompletion` com `.parsed` e `.refusal`]

### Pattern 3: `serviceRoleClient()` em `lib/supabase/server.ts`

**What:** Cliente Supabase que usa `SUPABASE_SERVICE_ROLE_KEY` em vez de anon key — bypassa RLS por design. Marcado com `import 'server-only'` para garantir que o bundler do Next nunca o coloque no client.

**When to use:** APENAS para operações server-side onde RLS é inviável (caso atual: o user pode estar deslogado — formulário público — e o enrichment precisa atualizar a row anyway).

**Example (a CRIAR em `lib/supabase/server.ts`):**
```typescript
// lib/supabase/server.ts (acrescentar ao final do arquivo existente)
import 'server-only'; // CRÍTICO — impede vazamento pro client bundle
import { createClient as createSupabaseClient } from '@supabase/supabase-js';
import type { SupabaseClient } from '@supabase/supabase-js';

 * Cliente Supabase com service role — bypassa RLS por design.
 *
 * USO RESTRITO:
 *   - Phase 7.6 ai/enrichment.ts: UPDATE pós-IA quando user pode estar deslogado
 *     (formulário público anônimo) + para evitar acoplamento ao JWT do user.

 * o build se este módulo entrar em um client bundle.

export function serviceRoleClient(): SupabaseClient<Database> {

  const key = process.env.SUPABASE_SERVICE_ROLE_KEY;

    throw new Error(
      '[supabase/server] SUPABASE_SERVICE_ROLE_KEY ou NEXT_PUBLIC_SUPABASE_URL ausente'

  }
```

### Anti-Patterns to Avoid

- **`import 'server-only'` ausente em `lib/ai/enrichment.ts` e na export `serviceRoleClient`** — sem isso, o tree-shaking pode falhar e a OPENAI_API_KEY / SERVICE_ROLE_KEY vazar para o bundle do cliente. SEMPRE adicionar.
- **Logar `tenant_id` raw em mensagem de erro persistida no DB ou em console.error** — risco de cross-tenant info leak via logs Vercel. Use `tenantId.slice(0,8)+'...'` ou simplesmente NÃO logue.
- **Interpolar `tenant_id`, `user_email`, IDs, ou qualquer info de tenant DENTRO do prompt OpenAI** — defesa anti-prompt-injection cross-tenant (AI-TEST-02). Só passar campos de processo/automação para o modelo. O prompt NUNCA precisa do tenant_id pra fazer seu trabalho.
- **Esquecer o `import 'server-only'` em `lib/ai/prompts.ts`** — não é estritamente necessário (não tem segredo), mas é boa prática para evitar que strings de prompt acabem no client bundle (~1KB+).
- **UPDATE sem o filtro `.eq('ai_enrichment_status', 'pending')`** — quebra idempotência. Se `after()` rodar 2x (ou catch-up job no futuro), sobrescreve dados editados pelo admin.
- **Persistir score calculado** — score continua sendo runtime via view. NÃO tocar em `opportunity_score()`.
- **Esquecer de fazer DROP da view antes de re-CREATE em migration 0010** — o `o.*` na view re-ordena colunas e Postgres bloqueia (`ERROR 42P16`). Padrão de 0008/0009.
- **Criar tabela `ai_enrichment_log` separada com tenant_id** — fora de escopo (deferred); migration 0010 só adiciona 3 colunas na `opportunities`.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Chamada OpenAI com retry + tipos | Raw `fetch('https://api.openai.com/v1/chat/completions')` | `openai` v6 SDK | SDK tem retry (2x default) + backoff + tipos + helpers. Já standard na indústria. |

## Common Pitfalls

### Pitfall 1: `gpt-4o-mini` retornando `refusal` em vez de `content`

**What goes wrong:** O modelo pode recusar a request (ex.: detecta conteúdo "sensível" no `processo` do usuário — improvável mas possível). `completion.choices[0].message.content` fica `null` e `completion.choices[0].message.refusal` vira uma string explicando.

**Why it happens:** OpenAI's safety policy filters; mais comum com user-generated content (formulário público).

**How to avoid:** Sempre checar `message.refusal` ANTES de checar `message.parsed`. Se houver refusal, escrever `ai_enrichment_status='failed'` com `ai_enrichment_error='refusal: <reason>'`. Admin edita manualmente.

**Warning signs:** Spike de `failed` no `ai_enrichment_status` correlacionado a tipos de processo específicos.

[VERIFIED: developers.openai.com/api/docs/guides/structured-outputs — "Access via choices[0].message.refusal (string or null)"]

### Pitfall 2: `gpt-4o-mini` esgotando tokens com `finish_reason='length'`

**What goes wrong:** O SDK joga `LengthFinishReasonError` se o output truncar. `message.parsed` fica indefinido.

**Why it happens:** Output muito longo + `max_tokens` default insuficiente. Com strict JSON, a resposta pode estourar se o user input for grande (lista de processos detalhados, etc.).

**How to avoid:** Setar `max_tokens: 800` explicitamente (o output esperado é ~300 tokens). Capturar `LengthFinishReasonError` no catch e escrever `failed`. Limitar tamanho de `formulario_extras` e `processo` que entram no prompt (já tem limit Zod max 2000 chars em `processo`).

**Warning signs:** `ai_enrichment_error` contém "LengthFinishReasonError" ou "finish_reason: length".

### Pitfall 3: JSON Schema strict requirements — todos os fields em `required`

**What goes wrong:** Zod permite campos opcionais (`.optional()`), mas OpenAI strict structured outputs **exige** que TODOS os campos do objeto estejam em `required`. Se algum for opcional, o helper `zodResponseFormat` pode gerar um schema rejeitado pela API com `Invalid schema for response_format`.

**Why it happens:** strict mode da OpenAI rejeita `additionalProperties != false` E exige `required` cobrir todas as chaves.

**How to avoid:** Para campos que conceitualmente podem ser "vazios" (ex.: `observacao` quando IA não tem nada a dizer), use `z.string()` que aceita string vazia em vez de `z.string().optional()`. Para arrays opcionais, use `z.array(...)` que aceita `[]`. NÃO use `.optional()` em campos top-level da resposta.

**Warning signs:** Erro 400 da OpenAI com mensagem `Invalid schema for response_format: required property 'X' is missing`.

[VERIFIED: developers.openai.com/api/docs/guides/structured-outputs — "All properties listed in 'required' array"]

### Pitfall 4: `after()` callback throwing → cold-restart drops work silently

**What goes wrong:** Se a Vercel function fizer cold-restart entre o `INSERT` retornar e o `after()` callback completar, o callback pode ser pulado sem log. Row fica em `pending` para sempre sem job de catch-up.

**Why it happens:** `after()` usa `waitUntil` que estende o lifecycle, MAS a Vercel pode terminar a instance após `maxDuration` ou em conditions específicos de scaling.

**How to avoid:** (a) NÃO depender de `after()` para correctness — o `ai_enrichment_status='pending'` permanece como flag para catch-up futuro (deferred backlog). (b) Garantir que `enrichOpportunity()` é completamente self-contained — não depende de variables capturadas que possam ter side effects pós-response. (c) Considerar adicionar tarefa em backlog 999.x: "Vercel Cron a cada 5min para re-enrichar `pending` mais antigos que 1h".

**Warning signs:** Query `SELECT count(*) FROM opportunities WHERE ai_enrichment_status='pending' AND created_at < now() - interval '1 hour'` retornando > 0.

[VERIFIED: nextjs.org/docs/app/api-reference/functions/after — "after will run for the platform's default or configured max duration of your route"]

### Pitfall 5: Tenant_id leak via prompt → cross-tenant info disclosure

**What goes wrong:** Dev coloca `tenant_id` no prompt "pra IA saber de qual cliente é". OpenAI loga o prompt (training? cache?) e cria risco de vazamento entre tenants — mesmo que small.

**Why it happens:** Confusão entre "context que ajuda a IA" e "info necessária pro raciocínio". Tenant_id é puramente para o UPDATE, não para o prompt.

**How to avoid:** Test snapshot em `tests/ai/prompts.test.ts` que assert que o prompt gerado NÃO contém UUIDs (regex pattern `/[0-9a-f]{8}-[0-9a-f]{4}-/i`). Code review explícito.

**Warning signs:** Grep por `tenant_id` em `lib/ai/prompts.ts` ou `buildEnrichmentPrompt(...)` — só pode aparecer em parâmetros desestruturados que NÃO entram no template literal.

### Pitfall 6: View `opportunities_with_score` quebra na migration 0010

**What goes wrong:** Adicionar 3 colunas (`ai_enrichment_status`, `ai_enrichment_error`, `ai_enriched_at`) em `opportunities` faz o `o.*` na view re-ordenar. `CREATE OR REPLACE VIEW` retorna `ERROR 42P16: cannot change name of view column "score" to "ai_enrichment_status"`.

**Why it happens:** Postgres bloqueia mudança da ordem das colunas em view replace.

**How to avoid:** Seguir padrão de migrations 0008 e 0009 — `drop view if exists opportunities_with_score; create view opportunities_with_score with (security_invoker = true) as select ...; grant select ... to authenticated;`.

**Warning signs:** Migration falha com `42P16` no apply manual.

### Pitfall 7: `lib/opportunities/queries.ts` whitelist NÃO inclui colunas novas

**What goes wrong:** `OPPORTUNITY_COLUMNS` em `queries.ts` é hardcoded. Após migration 0010, queries não retornam `ai_enrichment_status` → badge no modal nunca aparece.

**Why it happens:** Whitelist explícita foi um requisito do 7.5 Plan 05 (HARDEN-E-06) — segurança boa, mas exige update sempre que coluna nova é adicionada.

**How to avoid:** Plan deve listar explicitamente: "adicionar `ai_enrichment_status, ai_enrichment_error, ai_enriched_at` em `OPPORTUNITY_COLUMNS` em `lib/opportunities/queries.ts`".

**Warning signs:** Badge `ai_enrichment_status` no modal mostra "undefined" / não renderiza.

### Example: Migration 0010 SKETCH

```sql
-- supabase/migrations/0010_ai_enrichment.sql
-- =============================================================================
-- 0010_ai_enrichment.sql — Colunas de status de enriquecimento por IA
-- =============================================================================
-- Adiciona 3 colunas a `opportunities` para rastrear o pipeline assíncrono
-- de enriquecimento via OpenAI (Phase 7.6):
--   - ai_enrichment_status enum('pending','enriched','failed') default 'pending'
--   - ai_enrichment_error text null
--   - ai_enriched_at timestamptz null
--
-- A view `opportunities_with_score` precisa DROP + CREATE porque `o.*` na
-- view re-ordena as colunas (mesmo problema das migrations 0008/0009).
--
-- Pré-requisitos: 0001..0009 aplicadas.
-- =============================================================================

set session characteristics as transaction read write;
set default_transaction_read_only = off;
set check_function_bodies = off;

-- ---------------------------------------------------------------------------
-- 1. Enum + colunas + CHECKs
-- ---------------------------------------------------------------------------
do $$
begin
  if not exists (select 1 from pg_type where typname = 'ai_enrichment_status') then
    create type ai_enrichment_status as enum ('pending', 'enriched', 'failed');
  end if;
end$$;

alter table opportunities
  add column if not exists ai_enrichment_status ai_enrichment_status
    not null default 'pending',
  add column if not exists ai_enrichment_error  text,
  add column if not exists ai_enriched_at       timestamptz;

-- Length limit em error_message — defesa em profundidade
do $$
begin
  if not exists (
    select 1 from pg_constraint where conname = 'opportunities_ai_error_len_chk'
  ) then
    alter table opportunities
      add constraint opportunities_ai_error_len_chk
        check (ai_enrichment_error is null or length(ai_enrichment_error) <= 1000);
  end if;
end$$;

-- Índice parcial — queries de monitoring "quantas pending estão velhas?"
create index if not exists opportunities_ai_pending_idx
  on opportunities(created_at)
  where ai_enrichment_status = 'pending';

-- ---------------------------------------------------------------------------
-- 2. Re-cria view `opportunities_with_score` para incluir novas colunas
-- ---------------------------------------------------------------------------
drop view if exists opportunities_with_score;

create view opportunities_with_score
with (security_invoker = true) as
select
  o.*,
  opportunity_score(o.esforco, o.complexidade, o.tempo, o.objetivo) as score,
  case
    when opportunity_score(o.esforco, o.complexidade, o.tempo, o.objetivo) >= 70 then 'alta'
    when opportunity_score(o.esforco, o.complexidade, o.tempo, o.objetivo) >= 40 then 'media'
    else 'baixa'
  end as priority_level
from opportunities o;

grant select on opportunities_with_score to authenticated;

-- =============================================================================
-- FIM 0010 — ai_enrichment_status + error + enriched_at adicionados
-- =============================================================================
```

## Open Questions

1. **Quem pode editar os 9 campos enriquecidos no modal?**
   - What we know: Schema tem `profiles.role` mas o codebase pesquisado não mostrou enforcement em UI.
   - What's unclear: O CLAUDE.md fala "admin / cross-tenant fica para depois" — então TODO user autenticado de um tenant pode editar. Mas é isso que queremos para os campos da IA?
   - Recommendation: Planner adota "todo user autenticado do tenant edita" (paridade com edição existente) por simplicidade. Adicionar TODO `// TODO 999.x — restringir edição AI fields a role='coe-member' quando role enforcement entrar`.

2. **Como dar feedback de progresso no UI durante `pending` (3-8s)?**
   - What we know: O Server Action retorna `ok: true` instantaneamente. UI fecha o modal/redireciona.
   - What's unclear: Se o user voltar para a lista logo após criar, vai ver `pending` no badge da row recém-criada. UX disruption?
   - Recommendation: NÃO polling. Badge silencioso em `pending` (apenas para admin no modal). Se user terminar e fechar o modal, no próximo refresh já está `enriched`. Documentar como "feature, não bug" em comentário.

3. **Se o modelo `gpt-4o-mini` ficar offline temporariamente, retry?**
   - What we know: SDK retry 2x default com backoff exponential.
   - What's unclear: Após 2 retries, escrevemos `failed`. Catch-up job (futuro) re-tenta?
   - Recommendation: Aceitar `failed` definitivo para MVP. Adicionar TODO no backlog 999.x: "Vercel Cron re-enricha `pending` ou `failed` antigos".

4. **Idioma do prompt: EN ou PT?**
   - What we know: `gpt-4o-mini` aceita PT-BR fluentemente.
   - What's unclear: Para campos de saída em pt-BR (escopo, beneficios, observacao, risco), o prompt EN força o modelo a "traduzir" o user input para racicionar e voltar para pt-BR — pode causar perda de nuance.
   - Recommendation: SYSTEM prompt em **EN** (mais robusto e padrão indústria), explicitando "responda em pt-BR" para os campos textuais. User input fica em pt-BR como veio do formulário. **JÁ refletido na seção Code Examples.**

### Known Threat Patterns for OpenAI + Background Jobs

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Cross-tenant prompt leak (tenant_id no prompt) | Information Disclosure | NÃO interpolar tenant_id em prompts; snapshot test em prompts.test.ts. |
| Prompt injection via user input (processo, formulario_extras) | Tampering | Bound by structured output strict mode — modelo só pode responder no schema. User input em bloco delimitado. |
| Service role key vazando para client | Information Disclosure / Elevation | `import 'server-only'` em `lib/supabase/server.ts` serviceRoleClient(); audit grep em CI. |
| `ai_enrichment_error` contendo tenant_id ou PII | Information Disclosure | Sanitize message antes de persistir; truncar para 500 chars; check em prompts.test.ts. |
| OpenAI rate limit (429) → enrichment falha em massa | Denial of Service | SDK retry built-in; catch + `failed` log. Backlog 999.x: retry job. |
| Cold-restart drop silencioso → status='pending' órfão | Reliability | Monitoring query (Open Question 3); backlog 999.x catch-up job. |
