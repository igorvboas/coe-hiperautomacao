## Decisions Made

- **Option C lockada para tenant lookup público:** `serviceRoleClient()` em vez de anon client. Justificativa: RLS policy `tenants_select_own` (0001_init.sql:319-320) usa `id = current_tenant_id()` que requer `auth.uid()`. Session anônima do form público faz `current_tenant_id()` retornar NULL → query retornaria zero rows → `after()` nunca dispararia. Option B (mudar a RPC) descartada por quebrar a assinatura locked com migration 0007 (Plan 06 de 7.5). Verificação pré-flight feita via grep das policies em `supabase/migrations/` confirmando que NÃO existe policy anônima de SELECT em `tenants`.
- **PublicSubmitInput type mantido:** os 9 campos enriquecidos continuam opcionais no type, e o Server Action continua enviando defaults (`'medio'`, etc.) para a RPC. Plan 03 NÃO toca a RPC nem a assinatura — compat com 0007 preservada. IA depois sobrescreve via UPDATE em `enrichOpportunity`.
- **Mínimo touch no schema:** Apenas `esforco/complexidade/tempo/objetivo` mudados; os outros 5 campos (`ferramenta/escopo_automacao/beneficios_esperados/observacao/risco`) já eram opcionais ou tinham `.default([])` no baseline — sem alteração redundante.
- **`.strict()` preservado:** mass-assignment defense de 7.5 Plan 03 intacta. 18/18 testes em `tests/security/mass-assignment.test.ts` continuam verdes.

## Threat Mitigations Applied

| Threat ID | Mitigation aplicada |
| --------- | ------------------- |
| `T-07.6-C-01` (after() callback throw → response 500) | Callback envolto em try/catch defensivo. `enrichOpportunity` real (Plan 02) já tem catch interno; mas o wrapper aqui adiciona uma segunda camada e loga via `console.error` server-side. Test 3 valida: callback rejeitando NÃO propaga p/ response. |
| `T-07.6-C-02` (cookies/headers stale dentro do callback) | Closure captura primitivos (`oppId`, `tenantId`) ANTES do `after()` — callback NUNCA referencia `cookies()`/`headers()`/`req`. Validável via leitura do callback inline em `actions.ts`. |
| `T-07.6-C-03` (caller força `tenant_id` no payload) | `tenant_id` SEMPRE derivado server-side: createOpportunity usa `profile.tenant_id` (lookup via `auth.uid()`); createPublicOpportunity usa `tenants.id` resolvido por `slug` via service role. Schema `.strict()` de 7.5 Plan 03 rejeita `tenant_id` no input do user (defesa em camadas). |
| `T-07.6-C-04` (cold-restart pula after()) | Aceito conforme PHASE.md decisões locked. `ai_enrichment_status='pending'` permanece como flag p/ job catch-up futuro (backlog 999.x). |
| `T-07.6-C-05` (enrichment error vaza p/ response) | `after()` roda APÓS response HTTP. Erros logados via `console.error` apenas. Test 3 valida response continua `{ok:true,id}` mesmo com callback rejeitando. |
