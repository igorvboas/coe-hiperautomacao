    **CRÍTICO:** `next/server.after` precisa ser mockado de forma que possamos verificar (a) que foi chamado, (b) que o callback passado executa corretamente. Padrão:

    ```typescript
    const afterCalls: Array<() => Promise<void> | void> = [];
    vi.mock('next/server', () => ({
      after: vi.fn((cb: () => Promise<void> | void) => {
        afterCalls.push(cb);
      }),
    }));
    ```

    Assim em cada teste podemos: (a) assert `after` foi chamado N vezes, (b) executar `await afterCalls[0]()` para verificar que dispara `enrichOpportunity`.

    Structure básica:


## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action body → after() callback | callback executa pós-response; sem stack trace para o user |
| Server Action → enrichOpportunity (via after) | confiança transitiva — closure captura tenant_id resolvido server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07.6-C-01 | Denial of Service | after() callback throw → potencial impacto em response (na verdade não, after roda pós-response) | mitigate | try/catch envolve `await enrichOpportunity()` no callback; log estruturado em console.error; revalidatePath() chamado ANTES do after schedule |
| T-07.6-C-02 | Tampering | Capture de cookies/headers dentro de after() pode falhar (referência stale) | mitigate | Closure captura `oppId` e `tenantId` ANTES do after — não usa cookies/headers dentro do callback |
| T-07.6-C-03 | Spoofing | Caller (cliente) força enrichment ao injetar tenant_id no payload | mitigate | `tenant_id` derivado server-side de `profile.tenant_id` (createOpportunity) ou de `tenants.slug` resolvido server-side (createPublicOpportunity). Schema `.strict()` de 7.5 Plan 03 rejeita `tenant_id` no input do user (defesa em camadas). |
| T-07.6-C-04 | Reliability | Cold-restart pula after() callback → row órfã em pending | accept (documented) | `ai_enrichment_status='pending'` default permanece como flag para job catch-up futuro (deferred 999.x); admin pode editar manualmente |
| T-07.6-C-05 | Information Disclosure | Erro do enrichment vaza para response HTTP | mitigate | after() roda APÓS response; mesmo se callback throw, response já foi enviada como `{ok:true,id}`. Erros logados em `console.error` server-side (Vercel logs) e em `ai_enrichment_error` na row |

- `next/server.after` API mudar entre versões — atualmente stable desde Next 15.1; Next 16.x usado. Risco baixo.
