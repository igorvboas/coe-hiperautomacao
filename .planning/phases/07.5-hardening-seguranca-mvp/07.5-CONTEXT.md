# Phase 7.5: Hardening de Segurança MVP — Context

**Gathered:** 2026-05-21
**Status:** Ready for planning
**Source:** Revisão de segurança ad-hoc — checklist OWASP-style cruzado com o contexto multi-tenant + formulário público anônimo

<domain>
## Phase Boundary

Endurecer a plataforma contra vetores de ataque relevantes ao contexto **multi-tenant + formulário público** antes do deploy de produção (Phase 8). Foco em **defesa em profundidade** das decisões já tomadas (RLS, score calculado, sem admin) — não introduzir novos modelos de auth, não criar painel admin, não adicionar audit log universal.

Cinco blocos de trabalho:
- **A.** Testes automatizados de isolamento de tenant (RLS + IDOR)
- **B.** Validação centralizada com Zod (Mass Assignment + length limits)
- **C.** Atomicidade na geração de `seq_id` por tenant
- **D.** Hardening do formulário público (rate limit, captcha, limites de payload, logging)
- **E.** Auditoria de segredos + headers de segurança

</domain>

<decisions>
## Implementation Decisions

### Estratégia geral

- **Defesa em profundidade**: assumir que cada camada pode falhar. RLS (DB) + filtro explícito (server) + Zod (input) + headers (browser) trabalham juntos.
- **Falha segura**: erros de validação retornam 4xx genérico em pt-BR; nunca vazam detalhes internos (nomes de coluna, queries, stack traces).
- **Sem dependências exóticas**: usar primitivos já no projeto (Zod, Supabase, Vercel BotID/Turnstile). Não introduzir Redis dedicado para rate limit no MVP — usar Upstash via Vercel Marketplace só se BotID/middleware não bastar.

### Bloco A — Testes de isolamento

- **Test runner**: Vitest (já alinhado com Next.js 16). Adicionar como devDependency.
- **Estratégia**: testes de integração que falam com Supabase real (instância local via `supabase start` ou projeto de teste dedicado), **NÃO** mocks. Mock de DB mascara bug de RLS — não negociável.
- **Setup**: seed de dois tenants (`fgcoop` + `acme-test`) + usuário em cada. Cada teste autentica como um e tenta acessar dados do outro.
- **Cobertura mínima**: SELECT, INSERT, UPDATE, DELETE em `opportunities`, `opportunity_phases`, `users` (separados por verbo, separados por tabela).
- **CI**: rodar em GitHub Actions (ou pre-push hook) antes de qualquer deploy. Plan 8 (deploy) confia que estes testes passaram.

### Bloco B — Zod

- **Source of truth único**: schemas em `lib/opportunities/schema.ts`. Mesmo schema usado pelo form client (react-hook-form ou nativo) **e** pelo Server Action.
- **Whitelist explícita de campos**: `.strict()` no schema raiz — Zod rejeita campos não declarados. Bloqueia Mass Assignment.
- **`tenant_id`, `created_by`, `seq_id` NUNCA no schema do cliente** — derivados no server a partir de `auth.uid()`.
- **Limites**:
  - `processo`: max 2000 chars
  - `solicitante`, `email`, `area`, `subarea`: max 200 chars
  - `escopo_automacao[]`, `beneficios_esperados[]`: max 20 items × max 200 chars cada
  - `formulario_extras` jsonb: schema Zod tipado (não `z.any()`), max ~8KB serializado
- **Erro retornado ao cliente**: pt-BR, genérico ("Campo inválido"). Detalhe vai para log do server.

### Bloco C — Atomicidade `seq_id`

- **Diagnóstico primeiro**: inspecionar trigger atual em 0001_init.sql. Se for `select max(seq_id) + 1`, a race é real.
- **Solução preferida**: tabela auxiliar `tenant_sequences(tenant_id, last_seq)` + função `next_seq_id(p_tenant_id) returns int` com `update ... returning last_seq + 1` (atômico via lock de row).
- **Alternativa**: unique constraint `(tenant_id, seq_id)` + retry loop no insert (mais complexo, evitar).
- **Migration**: `0006_seq_id_atomic.sql` — backfill consistente, sem downtime.
- **Teste**: script paralelo (50 conexões inserindo no mesmo tenant) — todos devem ter `seq_id` único.

### Bloco D — Formulário público

- **Defesa anti-abuso em camadas (MVP)**:
  - Vercel BotID (classificação de bot, primeira linha)
  - Cloudflare Turnstile obrigatório (challenge humano, segunda linha)
  - Payload limits na RPC + Zod no Server Action (linha final, defesa em profundidade)
- **Rate limit por janela fixa = deferred**: 5/10min por IP e 50/hora por slug ficam para pós-MVP (Upstash Ratelimit via Vercel Marketplace ou Vercel WAF Rate Limit SDK). Motivo: BotID + Turnstile cobrem ~95% do vetor; janela fixa exige infra adicional (Redis) cujo custo só se justifica se monitoring mostrar abuso real. Documentado em PHASE.md "Out of scope" e em `<deferred>` deste CONTEXT.md.
- **CAPTCHA**: Cloudflare Turnstile (gratuito, sem captcha visual chato). Token validado server-side antes de chamar a RPC. Tornar **obrigatório** no `/r/[slug]`.
- **Limites de payload na RPC** ([0005_public_form_functions.sql](supabase/migrations/0005_public_form_functions.sql)):
  - Adicionar `if array_length(p_escopo_automacao, 1) > 20 then raise exception ...`
  - Mesmo para `p_beneficios_esperados`
  - Validar `length(p_formulario_extras::text) <= 8192`
  - Validar length max em `p_solicitante`, `p_processo`, `p_email` (defesa em profundidade — já validado no client mas pode ser bypassado)
- **Logging**: nova tabela `public_form_submissions(id, tenant_id, slug, ip, user_agent, status, error_message, created_at)`. Anonimizar IP após 30 dias (hash ou truncar octeto final). Rodar limpeza via cron Vercel ou trigger.
- **Defesa em profundidade contra storage exhaustion**: já que a função é `SECURITY DEFINER`, ela bypassa RLS. Toda validação dentro dela é a linha final de defesa.

### Bloco E — Segredos + headers

- **`SUPABASE_SERVICE_ROLE_KEY`**:
  - Grep no projeto: confirmar que aparece **apenas** em `lib/supabase/server.ts` e arquivos sem `'use client'`.
  - Confirmar configurado no Vercel via `vercel env` (não em `vercel.ts` literal).
  - Adicionar `.env.local` ao `.gitignore` (provavelmente já está — verificar).
- **Selects no client**: auditar `lib/opportunities/queries.ts` e qualquer call de `.select()` — substituir `select('*')` por colunas explícitas em rotas sensíveis (modal, listagem que volta para o usuário).
- **Headers de segurança** via `next.config.ts` `headers()` async ou em `proxy.ts`:
  - `Content-Security-Policy`: `default-src 'self'; script-src 'self' 'unsafe-inline' challenges.cloudflare.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self' *.supabase.co challenges.cloudflare.com; frame-src challenges.cloudflare.com` (ajustar com Turnstile)
  - `X-Frame-Options: DENY`
  - `X-Content-Type-Options: nosniff`
  - `Referrer-Policy: strict-origin-when-cross-origin`
  - `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
- **Permissions-Policy**: desligar APIs do browser que não usamos (`camera=(), microphone=(), geolocation=()`).

- Estrutura exata de `lib/security/*` — só criar se fizer sentido reutilizar.

<specifics>
## Specific Ideas

- **Captcha**: Cloudflare Turnstile (free, sem captcha visual). Site key + secret no Vercel env. Componente client-side carrega script de challenges.cloudflare.com.
- **Rate limit**: começar com Vercel BotID (zero infra). Se métricas mostrarem abuso, evoluir para Upstash Redis via Marketplace.
- **Tabela `public_form_submissions`**: schema sugerido:
  ```sql
  create table public_form_submissions (
    id uuid primary key default gen_random_uuid(),
    tenant_id uuid references tenants(id),
    slug text not null,
    ip_hash text not null,  -- hash desde o início, nunca IP raw
    user_agent text,
    status text not null,    -- 'success' | 'rate_limited' | 'invalid' | 'captcha_failed'
    error_message text,
    created_at timestamptz default now()
  );
  -- RLS: somente admins podem ler. No MVP sem admin: ninguém lê via cliente.
  ```
- **Estrutura de testes sugerida**:
  ```
  tests/
    security/
      tenant-isolation.test.ts    # RLS cruzado
      mass-assignment.test.ts     # Zod whitelist
      atomicity.test.ts           # seq_id race
      public-form.test.ts         # rate limit + captcha + payload limits
    setup/
      seed-test-tenants.ts        # cria fgcoop + acme-test
      supabase-test-client.ts     # client com auth de teste
  ```

</specifics>

<deferred>
## Deferred Ideas

- **Vercel WAF customizado** — usar defaults + BotID; rules customizadas só se houver evidência de abuso.
- **Audit log universal** (todas as ações de todos os usuários) — fora do MVP por decisão em CLAUDE.md.
- **Painel admin pra ver `public_form_submissions`** — pós-MVP. No MVP, query manual via SQL quando necessário.
- **2FA** — Supabase Auth suporta; ativar só se cliente piloto pedir.
- **Bug bounty / pen test externo** — pós-piloto.
- **Validação de uploads** — sem uploads no MVP.
- **Rate limit por janela fixa** (Upstash Ratelimit, 5/10min IP + 50/h slug) — pós-MVP. Critério de reativação: relatório de monitoring com taxa de abuso > X submissões/hora repetidas vindo do mesmo IP/slug por dia consecutivo.

</deferred>
