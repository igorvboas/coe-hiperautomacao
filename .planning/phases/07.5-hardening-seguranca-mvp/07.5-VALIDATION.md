# Phase 07.5 — Validation Strategy

> Per-phase validation contract for feedback sampling during execution. Derived from `07.5-RESEARCH.md` §Validation Architecture and the five-block scope (A–E) in `07.5-CONTEXT.md`.

---

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | Vitest 3.x (Wave 0 installs) |
| **Config file** | `vitest.config.ts` (Wave 0 creates) |
| **Quick run command** | `npm run test -- --pool=forks --poolOptions.forks.singleFork` |
| **Full suite command** | `npm run test:security` (Wave 0 adds script — runs Vitest + curl-based header checks + grep audit) |
| **Estimated runtime** | ~30–60s local against `supabase start` instance |

Notes:
- Vitest MUST run with `pool: 'forks'` and `singleFork: true` — concurrent test files against the same Supabase project cause FK violations when one test deletes a user another just inserted.
- Tests hit a real Supabase (local Docker via `supabase start` OR a dedicated test project). NEVER mocks of the DB layer — mocking defeats the RLS verification (decision locked in CONTEXT.md §Bloco A).
- Test env loaded via `.env.test` (or `.env.local` with `NEXT_PUBLIC_SUPABASE_URL` pointing to local). Service-role key NEVER imported by client code; tests use the anon key + tenant-scoped JWTs.

---

## Sampling Rate

- **After every task commit:** Run `npm run test -- <touched-file>` (target the test for the file just changed).
- **After every plan wave:** Run `npm run test:security` (full suite — RLS + mass-assignment + atomicity + public-form + headers + secret audit).
- **Before `/gsd-verify-work`:** Full suite green. `curl -I` of `/` and `/r/[slug]` shows all five locked headers. `grep -r "SUPABASE_SERVICE_ROLE_KEY" app/ components/` returns zero hits.
- **Max feedback latency:** ~60 seconds (full suite); ~5s (single file).
