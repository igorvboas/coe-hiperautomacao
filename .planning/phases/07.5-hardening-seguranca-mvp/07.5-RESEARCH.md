# Phase 7.5: Hardening de Segurança MVP — Research

**Researched:** 2026-05-21
**Domain:** AppSec defensivo para SaaS multi-tenant (Next.js 16 + Supabase) com endpoint público anônimo
**Confidence:** HIGH (a maioria das peças foi validada contra docs oficiais Next.js 16, Cloudflare Turnstile, Vercel BotID; algumas escolhas comportamentais marcadas como MEDIUM)

## Summary

Phase 7.5 endurece a plataforma em cinco frentes (A–E) antes do deploy de produção do piloto FGCoop. O trabalho NÃO introduz nova arquitetura — endurece o que já existe: a RLS já implementada em `0001_init.sql`, os Zod schemas em `lib/opportunities/schema.ts`, o trigger `set_opportunity_seq_id()` (race-prone hoje, vide §"Inspeção do trigger atual"), a RPC `create_public_opportunity` em `0005_public_form_functions.sql`, e o `proxy.ts` que já existe vazio para headers.

A pesquisa confirma que **todas as decisões locked no CONTEXT.md são executáveis com o stack atual** (Next.js 16.2.6, React 19.2.4, Zod 4.4.3, @supabase/ssr 0.10.3). A única dependência runtime nova obrigatória é `botid` (Vercel BotID) + `@marsidev/react-turnstile` (componente client React para Cloudflare Turnstile). Vitest, Upstash Ratelimit e libs de teste entram como devDependencies. Não há gap de versão. Não há feature em training data que tenha sido removida ou renomeada (Next 16 só renomeou middleware→proxy, o que já está aplicado).

**Recomendação primária:** Atacar nesta ordem — (1) **Bloco C** (atomicidade `seq_id`, migration isolada, testável sem mudar app code), (2) **Bloco B** (Zod `.strict()` + remover spread de `actions.ts`), (3) **Bloco A** (Vitest + seed dois tenants), (4) **Bloco E** (headers + auditoria), (5) **Bloco D** (Turnstile + BotID + logging table). Esta ordem minimiza retrabalho — C blinda o DB, B blinda o server, A produz a rede de testes, E é declarativo, D é a frente mais grande e se beneficia de tudo anterior.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Testes RLS de isolamento de tenant (Bloco A) | Database (Supabase local) | Test runner (Vitest, Node) | RLS vive no Postgres — só validável contra Postgres real. Mocks de DB mascaram o bug que estamos testando contra. |
| Validação de input centralizada (Bloco B) | Backend (Server Action) | Frontend (form client, mesmo schema) | Schema único compartilhado; client valida para UX, server valida para segurança (defesa em profundidade). |
| Atomicidade `seq_id` (Bloco C) | Database (função PL/pgSQL + tabela aux) | — | Concorrência resolve-se onde a concorrência acontece (DB). App não deve nem saber. |
| Rate limit por IP (Bloco D) | Edge / Proxy (Vercel BotID + opcional middleware em `proxy.ts`) | API/Server Action (`checkBotId()`) | BotID classifica edge; Server Action consome resultado. Rate limit por slug (50/h) é app-level, fica no Server Action via Upstash se BotID não bastar. |
| Captcha (Bloco D) | Frontend (widget Turnstile) | Backend (Server Action chama siteverify) | Token vem do client, validação é mandatória no server. |
| Logging de submissões públicas (Bloco D) | Database (tabela `public_form_submissions`) | Backend (Server Action insere antes/depois da RPC) | Persistência em Postgres, escrita via service role ou função `SECURITY DEFINER`. |
| Security headers (Bloco E) | Edge / Proxy (`proxy.ts` ou `next.config.ts headers()`) | — | Headers são response-level; já temos `proxy.ts` em uso, soma-se a ele. |
| Audit de segredos (Bloco E) | DevOps / processo | Code (grep) | Não há código a escrever — é verificação. Resultado vira teste de regressão. |

### Pattern 1: Server Action com Zod strict + tenant derivado do server (Bloco B)

**What:** Toda Server Action que muta `opportunities` segue este shape. Schema é `.strict()` (rejeita campos extras). `tenant_id`, `created_by`, `seq_id`, `id` JAMAIS aparecem no schema do cliente — vêm de `auth.getUser()` no server.

**When to use:** Todo Server Action em `lib/opportunities/actions.ts` que recebe input do cliente.

**Example:**

```typescript
// lib/opportunities/schema.ts
import { z } from 'zod';

// Campos derivados do servidor — NÃO aparecem no schema cliente
const SERVER_DERIVED = ['tenant_id', 'created_by', 'seq_id', 'id'] as const;

const baseClientSchema = z.object({
  solicitante: z.string().trim().min(2).max(200),
  email: z.string().email().or(z.literal('')).optional(),
  area: z.string().trim().min(2).max(200),
  subarea: z.string().max(200).optional().or(z.literal('')),
  processo: z.string().trim().min(3).max(2000),
  frequencia: z.string().max(60).optional().or(z.literal('')),
  volume_medio: z.string().max(60).optional().or(z.literal('')),
  tempo_execucao: z.string().max(60).optional().or(z.literal('')),
  num_pessoas: z.string().max(60).optional().or(z.literal('')),
  ferramenta: toolEnum.nullable().optional(),
  escopo_automacao: z.array(z.string().max(200)).max(20).default([]),
  beneficios_esperados: z.array(z.string().max(200)).max(20).default([]),
  esforco: effortEnum,
  complexidade: complexityEnum,
  tempo: timeBucketEnum,
  objetivo: z.number().int().min(1).max(5),
  status: statusEnum.default('novo'),
  responsavel: z.string().max(200).optional().or(z.literal('')),
  notas: z.string().max(2000).optional().or(z.literal('')),
});

export const opportunityClientSchema = z.discriminatedUnion('source', [
  baseClientSchema.extend({
    source: z.literal('persona'),
    persona_extras: personaExtrasSchema.optional(),
  }).strict(),   // ← bloqueia Mass Assignment
  baseClientSchema.extend({
    source: z.literal('formulario'),
    formulario_extras: formularioExtrasSchema.optional(),
  }).strict(),
]);
```

```typescript
// lib/opportunities/actions.ts
'use server';
import { opportunityClientSchema } from './schema';

export async function createOpportunity(input: unknown) {
  // 1. Validação estrita — rejeita campos extras (tenant_id, created_by, id...)
  const parsed = opportunityClientSchema.safeParse(input);
  if (!parsed.success) {
    return { ok: false as const, error: 'Dados inválidos.', fieldErrors: parsed.error.flatten().fieldErrors };
  }
  const data = parsed.data;  // ← garantido: zero campos não-whitelisted

  // 2. Identidade vem do servidor
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return { ok: false as const, error: 'Sessão expirada.' };

  const { data: profile } = await supabase
    .from('profiles')
    .select('tenant_id')
    .eq('id', user.id)
    .single();
  if (!profile) return { ok: false as const, error: 'Profile não encontrado.' };

  // 3. Insert — tenant_id e created_by VÊM DO SERVER, não do payload
  const { data: inserted, error } = await supabase
    .from('opportunities')
    .insert({
      ...data,
      tenant_id: profile.tenant_id,    // ← server-derived
      created_by: user.id,             // ← server-derived
      // seq_id intencionalmente omitido → trigger preenche
    })
    .select('id')
    .single();

  // 4. Defesa em profundidade: mesmo que algo bypassasse Zod, RLS WITH CHECK
  //    no INSERT bloqueia tenant_id forjado (current_tenant_id() != payload).
  if (error || !inserted) return { ok: false as const, error: 'Erro ao criar.' };
  return { ok: true as const, id: inserted.id };
}
```

**Why `.strict()`:** Sem `.strict()`, Zod faz parse passthrough — um cliente malicioso enviando `{ tenant_id: '<outro>', source: 'persona', ... }` teria o `tenant_id` ignorado pelo Zod mas potencialmente persistido se o action fizesse `insert({ ...data, tenant_id: profile.tenant_id })` E o cliente tivesse forçado `data` antes (impossível no fluxo acima — mas torna isso impossível por construção). `.strict()` falha o parse inteiro se houver campo extra. [VERIFIED: zod.dev — `.strict()` adds runtime check]

**Mass Assignment defense layers (4):**
1. Zod `.strict()` rejeita `tenant_id`, `created_by`, `seq_id`, `id` no input.
2. Server Action faz spread só de `parsed.data` (já filtrado) + campos derivados explicitamente.
3. RLS `WITH CHECK (tenant_id = current_tenant_id())` no INSERT — DB rejeita se cliente forjar mesmo assim.
4. Trigger `set_opportunity_seq_id` (após hardening) bloqueia `seq_id` enviado pelo cliente (`if new.seq_id is not null then ignore and override`).

### Pattern 2: Sequência atômica por tenant (Bloco C)

**What:** Função PL/pgSQL `next_seq_id(tenant_uuid)` atualiza uma tabela auxiliar com row-level lock e retorna o próximo valor. Transacional, sem gaps em rollback.

**When to use:** Triggered no `before insert` de `opportunities`. Substitui completamente `set_opportunity_seq_id()` atual.

**Why this instead of CREATE SEQUENCE:** `nextval()` do Postgres **NÃO faz rollback** quando a transação aborta — se um insert falhar por outro motivo depois do `nextval`, o número é perdido (gap). Para `seq_id` exibido ao usuário ("Oportunidade #5, #7, ..."), gaps são ruins. Tabela auxiliar com `UPDATE ... RETURNING` é transacional (rollback restaura `last_seq`). [VERIFIED: postgresql.org/docs/current/sql-createsequence.html — "nextval operation is never rolled back"]

**Example:**

```sql
-- supabase/migrations/0006_seq_id_atomic.sql

set session characteristics as transaction read write;
set default_transaction_read_only = off;
set check_function_bodies = off;

-- 1. Tabela auxiliar
create table tenant_sequences (
  tenant_id  uuid primary key references tenants(id) on delete cascade,
  last_seq   int  not null default 0
);

-- RLS opcional — esta tabela só é tocada via trigger SECURITY DEFINER,
-- mas habilitamos por consistência. Nenhuma policy → role authenticated não lê.
alter table tenant_sequences enable row level security;
-- service_role bypassa RLS por default

-- 2. Backfill com o estado atual
insert into tenant_sequences (tenant_id, last_seq)
select tenant_id, coalesce(max(seq_id), 0)
from opportunities
group by tenant_id
on conflict (tenant_id) do update set last_seq = excluded.last_seq;

-- Garante linha para tenants ainda sem oportunidades
insert into tenant_sequences (tenant_id, last_seq)
select id, 0 from tenants
on conflict do nothing;

-- 3. Função atômica
create or replace function next_seq_id(p_tenant_id uuid)
returns int
language plpgsql
security definer
set search_path = public
as $$
declare
  v_next int;
begin
  -- Upsert + row lock atômico
  insert into tenant_sequences (tenant_id, last_seq)
    values (p_tenant_id, 1)
  on conflict (tenant_id) do update
    set last_seq = tenant_sequences.last_seq + 1
  returning last_seq into v_next;
  return v_next;
end;
$$;

-- 4. Substitui o trigger antigo
drop trigger if exists trg_opportunities_seq_id on opportunities;
drop function if exists set_opportunity_seq_id();

create or replace function set_opportunity_seq_id()
returns trigger
language plpgsql
as $$
begin
  -- ALWAYS override — cliente não pode forjar seq_id
  new.seq_id := next_seq_id(new.tenant_id);
  return new;
end;
$$;

create trigger trg_opportunities_seq_id
  before insert on opportunities
  for each row execute function set_opportunity_seq_id();
```

**Why atomic:** `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` adquire um row-level lock no upsert. Outras transações que tentam o mesmo `tenant_id` aguardam (não fazem busy-wait — vai para `pg_locks`). Quando a primeira termina, a segunda lê o `last_seq` atualizado.

**Test:** Script com 50 conexões paralelas inserindo no mesmo tenant — todos devem ter `seq_id` único e contínuo (1..50).

```sql
-- tests/security/atomicity.test.ts pode invocar isto via raw SQL:
-- (em paralelo, 50 vezes) com tenant fgcoop
insert into opportunities (tenant_id, source, solicitante, area, processo)
values ($1, 'persona', 'concurrent', 'test', 'race test')
returning seq_id;
```

**Pitfall:** O trigger SEMPRE sobrescreve `new.seq_id` — não use `if new.seq_id is null then`. Cliente malicioso enviando `seq_id: 999999` seria respeitado de outro modo.

### Pattern 3: Cloudflare Turnstile + Server Action verify (Bloco D)

**What:** Widget invisível Turnstile no `PublicForm.tsx` gera token; Server Action chama `https://challenges.cloudflare.com/turnstile/v0/siteverify` com secret antes de bater na RPC.

**Endpoint:** `POST https://challenges.cloudflare.com/turnstile/v0/siteverify` — aceita JSON ou form-urlencoded. [VERIFIED: developers.cloudflare.com/turnstile/get-started/server-side-validation/]

**Request body:**
- `secret` (obrigatório) — secret key (env `TURNSTILE_SECRET_KEY`)
- `response` (obrigatório) — token vindo do widget no client
- `remoteip` (opcional) — IP do cliente (do header `x-forwarded-for`)
- `idempotency_key` (opcional) — UUID para retries seguros

**Response shape:**
```json
{
  "success": true,
  "challenge_ts": "2026-05-21T19:00:00.000Z",
  "hostname": "coe-fgcoop.example.com",
  "action": "public_form_submit",
  "cdata": "optional custom data",
  "error-codes": []
}
```

**Token validity:** 300 segundos (5 min), single-use. Reutilização → `error-codes: ['timeout-or-duplicate']`. [VERIFIED: docs Cloudflare]

**Example — helper:**

```typescript
// lib/security/turnstile.ts
import 'server-only';

const SITEVERIFY_URL = 'https://challenges.cloudflare.com/turnstile/v0/siteverify';

export type TurnstileResult =
  | { ok: true; hostname: string; action: string | null }
  | { ok: false; errorCodes: string[] };

export async function verifyTurnstileToken(
  token: string,
  remoteIp?: string,
): Promise<TurnstileResult> {
  const secret = process.env.TURNSTILE_SECRET_KEY;
  if (!secret) throw new Error('TURNSTILE_SECRET_KEY ausente no env');

  const body = new URLSearchParams();
  body.append('secret', secret);
  body.append('response', token);
  if (remoteIp) body.append('remoteip', remoteIp);
  body.append('idempotency_key', crypto.randomUUID());

  const res = await fetch(SITEVERIFY_URL, {
    method: 'POST',
    body,
    // Não enviar credentials, não cachear:
    cache: 'no-store',
  });

  if (!res.ok) {
    return { ok: false, errorCodes: ['network-error'] };
  }

  const data = (await res.json()) as {
    success: boolean;
    hostname?: string;
    action?: string;
    'error-codes'?: string[];
  };

  if (!data.success) {
    return { ok: false, errorCodes: data['error-codes'] ?? ['unknown'] };
  }

  return {
    ok: true,
    hostname: data.hostname ?? '',
    action: data.action ?? null,
  };
}
```

### Pattern 6: Client IP em Vercel (helper)

**What:** Vercel sobrescreve `x-forwarded-for` para evitar spoofing. O primeiro IP da lista (left-most) é o cliente real.

```typescript
// lib/security/client-ip.ts
import 'server-only';
import { headers } from 'next/headers';

export async function getClientIp(): Promise<string> {
  const h = await headers();
  const fwd = h.get('x-forwarded-for') ?? '';
  // Vercel sempre escreve só o IP do cliente — mas em outros provedores
  // pode vir "client, proxy1, proxy2". Pegar o primeiro:
  const first = fwd.split(',')[0]?.trim();
  return first || h.get('x-real-ip') || '0.0.0.0';
}
```

[VERIFIED: vercel.com/docs/headers/request-headers — Vercel reescreve x-forwarded-for com o IP do cliente para impedir spoofing]

### Pattern 7: Hash de IP (Bloco D logging)

**What:** Tabela `public_form_submissions` NUNCA armazena IP raw. Hash SHA-256 com salt vindo do env (não da request).

```typescript
// lib/security/hash-ip.ts
import 'server-only';
import { createHash } from 'node:crypto';

export function hashIp(ip: string): string {
  const salt = process.env.IP_HASH_SALT;
  if (!salt) throw new Error('IP_HASH_SALT ausente');
  return createHash('sha256').update(`${salt}:${ip}`).digest('hex');
}
```

**Why salt:** Sem salt, hash de IP é tabela-rainbow trivial (IPv4 cabe em < 4 bilhões de hashes precomputados). Com salt secreto, hash vira pseudonimização real. Salt fica em `IP_HASH_SALT` no Vercel env (não em git).

### Anti-Patterns to Avoid

- **`select('*')` em rotas autenticadas:** retorna colunas que migrations futuras adicionem (vazamento por padrão). **Substituir por whitelist explícita.** Auditados em `lib/opportunities/queries.ts` linhas 24, 108, 130 — todos são `opportunities_with_score` que é view do projeto, **MEDIUM risk** (não é tabela arbitrária). Mesmo assim, recomendado trocar por colunas explícitas para defesa em profundidade.
- **Spread de input do cliente:** `supabase.insert({ ...input })` sem Zod `.strict()` antes. Já evitado em `actions.ts` atual, mas Zod sem `.strict()` permite que campos extras "passem" silenciosamente sem ir parar no DB (porque o spread é controlado) — não é Mass Assignment hoje, mas é frágil. **Adicionar `.strict()` torna o erro explícito.**
- **`'unsafe-inline'` em `script-src` em produção:** desabilita CSP efetivamente. Em dev é tolerado (Next.js usa eval/inline). Em produção, usar nonce. [VERIFIED: nextjs.org/docs/app/guides/content-security-policy]
- **Mock de DB em testes RLS:** falsa segurança. Toda lib que mocks `supabase.from` mente sobre policies do Postgres. **Sempre testar contra `supabase start` local.** [VERIFIED: supabase.com/docs/guides/local-development/testing/overview]
- **`SECURITY DEFINER` sem validação interna:** `create_public_opportunity` bypassa RLS. Toda validação dentro da função é a única defesa. **Hardening manda apertar checks lá dentro (length limits, array length, jsonb size).**
- **Token Turnstile reutilizado:** tokens são single-use. Reset do widget após cada submit, ou siteverify retorna `timeout-or-duplicate`. Sempre `turnstileRef.current?.reset()` após resposta. [VERIFIED: docs Cloudflare]
- **`nextval()` para `seq_id` user-facing:** gera gaps em rollback. Usar tabela auxiliar com `UPDATE ... RETURNING`. [VERIFIED: postgresql.org/docs/current/sql-createsequence.html]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Bot detection edge-side | "is mobile / is curl" heuristics | Vercel BotID (`checkBotId()`) | ML-trained classifier mantido pela Vercel; nossa heuristic vai vazar headless browsers reais. |
| Captcha | Honeypot fields, math questions | Cloudflare Turnstile | Bots resolvem honeypots; Turnstile usa fingerprinting + proof-of-work invisível. |
| Rate limit em memória | `Map<ip, count>` no Server Action | Upstash Ratelimit ou Vercel WAF Rate Limiting SDK | Server Actions na Vercel rodam em serverless: cada instância tem memória própria → contador zera entre requests. **Precisa estado compartilhado.** |
| CSP nonce | Hard-coded `nonce-xyz` | Gerar a cada request em `proxy.ts` via `crypto.randomUUID()` + base64 | Nonce previsível = inútil. [VERIFIED: nextjs.org docs] |
| IP detection | `req.connection.remoteAddress` | `headers().get('x-forwarded-for')` (primeiro IP) | Em serverless/edge, `remoteAddress` é o load balancer, não o cliente. Vercel reescreve `x-forwarded-for` para o cliente real. |
| Atomic sequence per-tenant | App-level counter Redis | Postgres function com `UPDATE ... RETURNING` em tabela aux | Transação Postgres é nossa fonte da verdade; Redis adicional adiciona modo de falha. Postgres `UPDATE ... RETURNING` em row específica é tão rápido quanto Redis SETNX para nosso volume. |
| Mass Assignment defense | Filtro manual por allowlist | Zod `.strict()` | Zod já é dependency; `.strict()` é uma palavra. |
| Captcha siteverify retries | Custom retry com backoff | Não retry — token é single-use | Retry no siteverify dispara `timeout-or-duplicate`. Cliente refaz o challenge. |

**Key insight:** O cenário "form público anônimo + multi-tenant" é exatamente o que Vercel BotID + Turnstile + Upstash Ratelimit foram desenhados para resolver. Tentar criar primitivos próprios neste MVP é puro waste — equipe pequena, sem AppSec dedicado.

## Common Pitfalls

### Pitfall 1: BotID retorna `isBot: false` localmente — sensação falsa de segurança

**What goes wrong:** Desenvolvedor adiciona `checkBotId()`, testa localmente, "funciona". Em produção, BotID dispara em uma das primeiras requests reais e rejeita o usuário do piloto.

**Why it happens:** Doc Vercel: "Local development always returns `isBot: false` unless you configure the `developmentOptions` option." [VERIFIED]

**How to avoid:** No teste manual com `vercel dev` (que simula edge) ou em preview deploy, validar fluxo end-to-end com Turnstile real. Adicionar nota nas tasks: "validar em preview deploy, não só local."

**Warning signs:** `verification.isBot === false` em 100% das requests locais.

### Pitfall 2: Turnstile token expira durante UX longo

**What goes wrong:** Wizard de 6 steps pode levar mais de 5 minutos para preencher. Token Turnstile expira → submit final falha com `timeout-or-duplicate`.

**Why it happens:** Tokens são válidos por 300s e single-use. [VERIFIED]

**How to avoid:** Gerar o token NO submit final, não na entrada do form. Widget Turnstile em modo `interaction-only` ou `execute()` manual disparado no clique do botão "Enviar".

**Warning signs:** Reports de "verificação anti-bot falhou" em formulários cujos usuários demoram >5 min.

### Pitfall 3: CSP nonce + Partial Prerendering / ISR são incompatíveis

**What goes wrong:** Adicionar nonce-based CSP em `proxy.ts` desliga static optimization. Páginas que estavam estáticas (ex: `/r/[slug]` se fosse cacheada) viram dinâmicas por força. [VERIFIED: nextjs.org docs]

**Why it happens:** Nonce muda a cada request — incompatível com builds prerendered.

**How to avoid:** Para esta phase, aceitar dinâmico (todas nossas páginas já são — `app/(app)/*` lê auth, `/r/[slug]` lê tenant). Se Phase 8 quiser CDN caching, voltar e considerar SRI (Subresource Integrity) experimental ou CSP sem nonce com `'unsafe-inline'` controlado.

**Warning signs:** Build warnings sobre "page can no longer be statically rendered".

### Pitfall 4: Vitest paralelismo + Supabase = flaky tests

**What goes wrong:** Vitest roda specs em paralelo por default. Testes de RLS deletam e recriam usuários em `beforeEach`; em paralelo, um teste deleta o user que outro acabou de criar → FK errors.

**Why it happens:** `beforeEach` global de cleanup + threads paralelas. [VERIFIED via supabase blog + index.garden]

**How to avoid:**
```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    pool: 'forks',
    poolOptions: { forks: { singleFork: true } },  // ← serializa
    sequence: { concurrent: false },
    setupFiles: ['./tests/setup/globalSetup.ts'],
    environment: 'node',
  },
});
```

**Warning signs:** Testes passam isoladamente, falham em suite. Erros de FK constraint ou `auth.users` not found.

### Pitfall 5: Trigger `set_opportunity_seq_id` antigo continua ativo após migration

**What goes wrong:** A migration `0006` cria a nova função mas esquece de fazer `drop trigger`. Postgres tem dois triggers no `before insert` — comportamento indeterminado (ambos rodam, último vence).

**Why it happens:** `create or replace function` substitui função; `create trigger` SEM `drop` cria duplicado.

**How to avoid:** A migration `0006_seq_id_atomic.sql` faz `drop trigger if exists trg_opportunities_seq_id on opportunities; drop function if exists set_opportunity_seq_id();` ANTES de recriar.

**Warning signs:** Após migration, `seq_id` pula valores (1, 3, 5...) ou repete.

### Pitfall 6: Service role key em código `'use client'` por descuido

**What goes wrong:** Em refactor, alguém importa `lib/supabase/server.ts` em arquivo `'use client'` — bundler embute service key no JS do navegador. Catástrofe.

**Why it happens:** O bundler do Next 16 inlina env vars `NEXT_PUBLIC_*` em client, mas vars sem prefixo (como `SUPABASE_SERVICE_ROLE_KEY` se algum dia existir) viram referência undefined em client — quebra silenciosamente OU pior, se alguém prefixar errado, expõe.

**How to avoid:**
1. Hoje **não há** `SUPABASE_SERVICE_ROLE_KEY` no código (grep confirma vazio).
2. Manter a regra: zero arquivos client importam `lib/supabase/server.ts`.
3. Teste de regressão: `npm run audit:secrets` que faz grep e falha CI.

**Warning signs:** Build output mencionando server-only module em client bundle; erro `process.env.X is undefined` em browser.

### Pitfall 7: select('*') vaza colunas novas

**What goes wrong:** Migration futura adiciona coluna `internal_pricing_notes` na `opportunities`. `fetchOpportunities` faz `select('*')` → coluna vai para o front, aparece no DOM ou DevTools.

**Why it happens:** `*` é "tudo o que existe agora E no futuro".

**How to avoid:** Whitelist de colunas em cada query (fora do escopo crítico desta phase mas vale documentar para Wave de cleanup).

**Warning signs:** Front quebra ou exibe colunas inesperadas após migration.

```typescript
  upgrade-insecure-requests;

const securityHeaders: Record<string, string> = {
  'Content-Security-Policy': csp,
  'X-Frame-Options': 'DENY',
  'X-Content-Type-Options': 'nosniff',
  'Referrer-Policy': 'strict-origin-when-cross-origin',
  'Strict-Transport-Security': 'max-age=63072000; includeSubDomains; preload',
  'Permissions-Policy': 'camera=(), microphone=(), geolocation=(), interest-cohort=()',
};

export async function proxy(request: NextRequest) {
  // 1. Update Supabase session (lógica que já existe)
  const response = await updateSession(request);

  // 2. Append security headers
  for (const [k, v] of Object.entries(securityHeaders)) {
    response.headers.set(k, v);
  }

  return response;
}

export const config = {
  matcher: [
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp)$).*)',
  ],
};
```

**CSP whys:**
- `connect-src` permite Supabase REST (`https://*.supabase.co`) e Realtime (`wss://*.supabase.co`).
- `frame-src` + `script-src` permitem o iframe Turnstile (`challenges.cloudflare.com`).
- `script-src 'unsafe-inline'` é necessário para Next.js inline scripts em produção SEM nonce (Next.js injeta hydration scripts inline). Alternativa "limpa" exige nonce-based CSP (vide Pitfall 3).
- `frame-ancestors 'none'` complementa `X-Frame-Options: DENY` (este é legacy; navegadores modernos usam `frame-ancestors`).

### Known Threat Patterns for Next.js + Supabase

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Mass Assignment via Server Action (cliente envia `tenant_id` forjado) | Tampering / Elevation of Privilege | Zod `.strict()` + RLS WITH CHECK |
| IDOR cross-tenant (cliente pede `/opportunities/<uuid-do-outro-tenant>`) | Information Disclosure | RLS SELECT policy filtra automaticamente; backend não precisa filtrar manualmente, mas o teste prova |
| SQL Injection | Tampering | Supabase client parametriza queries; nenhuma string interpolation crua. Único ponto sensível é `q.or` em `queries.ts:34` — JÁ tem escape de `%`, está OK. Single-quote ainda permitido — confirmar que parser do Supabase escapa |
| Public form abuse (spam, storage exhaustion) | Denial of Service | BotID + Turnstile + length limits em RPC + rate limit Upstash/WAF |
| Service role key leakage (cliente vê em bundle) | Information Disclosure | Audit grep; `lib/supabase/server.ts` é o único arquivo que importa service role; nunca importado em `'use client'` |
| Token Turnstile replay | Spoofing | Tokens single-use + 5 min TTL (Cloudflare enforce) + idempotency_key UUID |
| Race condition em `seq_id` (dois usuários, mesmo tenant, número duplicado visível) | Tampering | `next_seq_id()` com row-lock |
| Clickjacking (iframe da app em site malicioso) | Tampering | `X-Frame-Options: DENY` + CSP `frame-ancestors 'none'` |
| MITM downgrade attack | Information Disclosure | HSTS + `upgrade-insecure-requests` |
| XSS via campo `processo` etc. | Tampering | React escapa por default; CSP `script-src 'self'` bloqueia injeção mesmo se escapar passar; sem `dangerouslySetInnerHTML` no projeto (confirmar com grep) |
| Enumeration de tenants via slug | Information Disclosure | `fetch_public_tenant` retorna `null` para slug inexistente (não vaza "existe mas está suspenso") |

## Open Questions (RESOLVED)

1. **CSP com nonce ou sem?**
   - What we know: nonce dá CSP mais strict mas força dynamic rendering em tudo (Pitfall 3). Sem nonce, `'unsafe-inline'` precisa ficar em script-src.
   - What's unclear: cliente piloto FGCoop tem requirement de compliance que exige CSP strict (sem `unsafe-inline`)?
   - **RESOLVED:** **começar sem nonce (`'unsafe-inline'`)**, documentar como tech debt em PROJECT.md, apertar em phase futura se cliente solicitar.

2. **Vercel Rate Limit SDK vs Upstash Ratelimit?**
   - What we know: Vercel WAF Rate Limiting SDK (vercel.com/docs/vercel-firewall/vercel-waf/rate-limiting-sdk) chegou em 2024-2025 e é mais novo. Upstash Ratelimit é o padrão de fato do ecossistema há mais tempo.
   - What's unclear: o Rate Limiting SDK funciona com Server Actions ou só com Route Handlers? Custo?
   - **RESOLVED:** Vercel BotID + Cloudflare Turnstile + payload limits constitute the MVP anti-abuse defense. Rate limit by fixed window (5/10min IP, 50/h slug) is explicitly deferred to post-MVP backlog. Decision locked by user 2026-05-21. See PHASE.md "Out of scope" and CONTEXT.md `<deferred>`. No `lib/security/rate-limit.ts` is implemented in this phase.

3. **Onde insertar a row de log? Antes ou depois da RPC?**
   - What we know: CONTEXT.md sugere status `pending` antes, `success`/`error` depois.
   - What's unclear: se a RPC falhar de forma catastrófica (timeout), a row fica em pending para sempre.
   - **RESOLVED:** insert pending antes, update success/error depois; adicionar cron (Vercel) que marca `pending` mais antigos que 5min como `invalid` com error="timeout" (out-of-scope phase 7.5; nota para phase futura).

4. **Os testes RLS precisam de instância Supabase local OU podemos usar projeto Supabase remoto de testes?**
   - What we know: docs Supabase recomendam local (`supabase start`).
   - What's unclear: time tem laptop com Docker rodando o Supabase local? CI (Github Actions futura) também precisa.
   - **RESOLVED:** começar local (`supabase start`); planner deixa CI como `--skip-tests`-friendly até GitHub Actions ser configurada (out of scope desta phase, dela a Phase 8 cuida).

5. **`select('*')` em `lib/opportunities/queries.ts` deve mudar nesta phase?**
   - What we know: `select('*')` aparece 3 vezes, todas contra a view `opportunities_with_score`. View tem colunas controladas; risco baixo.
   - What's unclear: cliente piloto vai ter mais colunas adicionadas em phase futura que NÃO devem vazar.
   - **RESOLVED:** In scope as Plan 05 Task 2 (HARDEN-E-06). PATTERNS.md confirmed `select('*')` exists 3× in `lib/opportunities/queries.ts:24,108,130` against the controlled `opportunities_with_score` view — risk LOW but whitelisting is a cheap win in this phase.
