# Plan 07.5-06 — Migration Handoff (Supabase Cloud)

**Status:** pronto para aplicação manual.
**Arquivo da migration:** [`supabase/migrations/0007_public_form_hardening.sql`](../../../supabase/migrations/0007_public_form_hardening.sql)

Este documento substitui o passo `[BLOCKING]` original do plano (`supabase db push` / `supabase migration up`), porque o projeto roda em **Supabase Cloud (hosted)** e o usuário aplica a migration manualmente via **Dashboard SQL Editor**. Mesmo padrão usado em Plan 02 (handoff 0006).

---

## 1. Por que manual e não `supabase db push`

- Banco do projeto piloto vive em Supabase Cloud (não em instância local Docker).
- Nenhuma sessão `supabase link` foi feita neste worktree; rodar `supabase db push` exigiria `SUPABASE_ACCESS_TOKEN` + project ref e modificaria o DB remoto fora do controle visual do usuário.
- O usuário pediu controle explícito sobre quando o schema muda. Solução: agente prepara o SQL, usuário cola no Dashboard.

## 2. Pré-requisitos

- Migrations `0001_init.sql` ... `0006_seq_id_atomic.sql` já aplicadas no projeto Cloud.
  - **Atenção:** `0006_seq_id_atomic.sql` é da Wave 1 deste mesmo Phase 7.5 — se ainda não foi aplicada, ver [`07.5-02-MIGRATION-HANDOFF.md`](07.5-02-MIGRATION-HANDOFF.md) primeiro.
- Plan 06 NÃO toca em `tenant_sequences` nem no trigger `trg_opportunities_seq_id` (esses são do Plan 02). 0007 só adiciona `public_form_submissions` + funções + recria `create_public_opportunity`.

## 3. Como aplicar

### Passo a passo

1. Abrir o Supabase Dashboard do projeto → **SQL Editor** → **New query**.
2. Copiar o conteúdo INTEIRO de [`supabase/migrations/0007_public_form_hardening.sql`](../../../supabase/migrations/0007_public_form_hardening.sql) (~268 linhas).
3. Colar no editor.
4. Clicar **Run** (ou `Ctrl/Cmd + Enter`).
5. Esperar mensagem `Success. No rows returned`.

A migration é idempotente em parte (`create table if not exists`, `create or replace function`, `create index if not exists`) — rodar duas vezes não quebra, mas só precisa ser aplicada uma vez.

```sql
-- f) Smoke test rápido: log + update funcionam
-- ATENÇÃO: este insert cria 1 linha real em public_form_submissions (cleanup abaixo).
do $$
declare
  v_log_id uuid;
  v_status text;
begin
  -- log pending
  select log_public_form_attempt('fgcoop', 'a' || repeat('0',63), 'sql-smoke-test') into v_log_id;
  raise notice 'logged id: %', v_log_id;

  -- update para success
  perform update_public_form_attempt(v_log_id, 'success', null);

  -- confirma
  select status into v_status from public_form_submissions where id = v_log_id;
  raise notice 'status final: %', v_status;
  if v_status <> 'success' then
    raise exception 'smoke test FALHOU: status=%', v_status;
  end if;

  -- cleanup
  delete from public_form_submissions where id = v_log_id;
end $$;
-- Esperado: dois NOTICES (logged id + status final: success), sem exception
```

```sql
-- g) Smoke test dos payload limits: tentar criar com array > 20 deve falhar
-- ATENÇÃO: trocar 'fgcoop' por slug REAL existente no Cloud.
do $$
begin
  perform create_public_opportunity(
    'fgcoop',
    'sql-limit-test',
    'limit@test.local',
    'TI', '',
    'attack via array',
    '', '', '', '',
    '',
    array(select 'x' from generate_series(1, 25)),  -- 25 itens > 20
    array[]::text[],
    'medio', 'medio', 'medio',
    3::smallint,
    null
  );
  raise exception 'PROBLEMA: RPC aceitou 25 itens';
exception
  when others then
    if sqlerrm like '%escopo_automacao excede 20%' then
      raise notice 'OK: limits enforced — %', sqlerrm;
    else

-- Esperado: NOTICE "OK: limits enforced — escopo_automacao excede 20 itens"
```

Se todas as queries (a..g) retornarem o esperado, a migration está aplicada com sucesso.

## 4. Regenerar types do TypeScript (opcional, recomendado)

Depois que o schema mudar no Cloud, regenere `lib/database.types.ts` para incluir a tabela `public_form_submissions` e as funções novas:

```bash
npm run gen:types
```

Plan 06 NÃO depende dos types regenerados em tempo de execução (o Server Action chama `supabase.rpc('log_public_form_attempt', ...)` com nomes string). Mas os testes de integração em `tests/security/public-form.test.ts` usam `.from('public_form_submissions').select(...)`, então sem types regenerados pode haver `any` implícito no resultado (não bloqueia).

## 6. Configuração de env vars (faltam no Vercel — fora desta migration)

Após aplicar a migration, **três env vars precisam estar setadas** no Vercel para o formulário público funcionar end-to-end (Plan 06 deliverable, fora do escopo do SQL):

| Variável | Onde obter | Setar via |
|----------|------------|-----------|
| `NEXT_PUBLIC_TURNSTILE_SITE_KEY` | Cloudflare Dashboard → Turnstile → criar widget Mode: Invisible | `vercel env add NEXT_PUBLIC_TURNSTILE_SITE_KEY production preview` |
| `TURNSTILE_SECRET_KEY` | Mesmo widget → Secret key | `vercel env add TURNSTILE_SECRET_KEY production preview` |
| `IP_HASH_SALT` | `openssl rand -hex 32` | `vercel env add IP_HASH_SALT production preview` |

Sem `IP_HASH_SALT`, o `hashIp()` em `lib/security/hash-ip.ts` lança exception (defesa: nunca persistir IP sem salt). O Server Action captura e retorna mensagem pt-BR genérica ao cliente.

Sem `TURNSTILE_SECRET_KEY`, `verifyTurnstileToken()` retorna `ok: false` com `errorCodes: ['missing-secret-config']`. O Server Action captura e bloqueia o submit.

Sem `NEXT_PUBLIC_TURNSTILE_SITE_KEY`, o `page.tsx` do `/r/[slug]` lança erro em runtime (escolha deliberada: melhor falhar visível do que renderizar form quebrado).
