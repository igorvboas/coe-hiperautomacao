# Plan 07.5-02 — Migration Handoff (Supabase Cloud)

**Status:** pronto para aplicação manual.
**Arquivo da migration:** [`supabase/migrations/0006_seq_id_atomic.sql`](../../../supabase/migrations/0006_seq_id_atomic.sql)

Este documento substitui o passo `[BLOCKING]` original do plano (`supabase db push`), porque o projeto roda em **Supabase Cloud (hosted)** e o usuário aplica a migration manualmente via **Dashboard SQL Editor**.

---

## 1. Por que manual e não `supabase db push`

- Banco do projeto piloto vive em Supabase Cloud (não em instância local Docker).
- Nenhuma sessão `supabase link` foi feita neste worktree; rodar `supabase db push` exigiria `SUPABASE_ACCESS_TOKEN` + project ref e modificaria o DB remoto fora do controle visual do usuário.
- O usuário pediu controle explícito sobre quando o schema muda. Solução: agente prepara o SQL, usuário cola no Dashboard.

## 2. Pré-requisitos

- A migration `0001_init.sql` (schema base) já foi aplicada no projeto Cloud (Phases 1–7 já rodaram com ele).
- Nenhuma das migrations 0006+ foi aplicada antes — este é o primeiro item da numeração 0006.

## 3. Como aplicar

### Passo a passo

1. Abrir o Supabase Dashboard do projeto → **SQL Editor** → **New query**.
2. Copiar o conteúdo INTEIRO de [`supabase/migrations/0006_seq_id_atomic.sql`](../../../supabase/migrations/0006_seq_id_atomic.sql) (117 linhas).
3. Colar no editor.
4. Clicar **Run** (ou `Ctrl/Cmd + Enter`).
5. Esperar mensagem `Success. No rows returned`.

A migration é idempotente em parte (`create table if not exists`, `on conflict ... do update/nothing`, `create or replace function`, `drop trigger if exists`) — rodar duas vezes não quebra. Mas só precisa ser aplicada uma vez.

### Verificação pós-apply (no mesmo SQL Editor)

Rode estas queries DEPOIS de aplicar para confirmar que ficou tudo certo:

```sql
-- a) Tabela tenant_sequences existe, com PK, FK e RLS habilitada
select c.relname as table_name,
       c.relrowsecurity as rls_enabled,
       (select count(*) from pg_policies p where p.tablename = c.relname) as policy_count
from pg_class c
where c.relname = 'tenant_sequences';
-- Esperado: 1 linha, rls_enabled = true, policy_count = 0
```

```sql
-- b) Função next_seq_id existe e é SECURITY DEFINER
select p.proname,
       p.prosecdef as security_definer,
       pg_get_function_identity_arguments(p.oid) as args
from pg_proc p
where p.proname = 'next_seq_id';
-- Esperado: next_seq_id | security_definer=true | args="p_tenant_id uuid"
```

```sql
-- c) Trigger trg_opportunities_seq_id está ativo em opportunities
select tgname, tgenabled
from pg_trigger
where tgrelid = 'opportunities'::regclass
  and tgname = 'trg_opportunities_seq_id';
-- Esperado: tgname=trg_opportunities_seq_id | tgenabled='O' (origin/enabled)
```

```sql
-- d) Backfill: tenant_sequences tem 1 linha por tenant, last_seq alinhado com opportunities
select ts.tenant_id,
       t.slug,
       ts.last_seq,
       (select coalesce(max(seq_id), 0) from opportunities o where o.tenant_id = ts.tenant_id) as max_opp_seq
from tenant_sequences ts
join tenants t on t.id = ts.tenant_id
order by t.slug;
-- Esperado: para cada tenant, last_seq = max_opp_seq (e ambos = 0 para tenants vazios)
```

## 4. Regenerar types do TypeScript (opcional, recomendado)

Depois que o schema mudar no Cloud, regenere `lib/database.types.ts` para incluir a nova tabela `tenant_sequences` e a função `next_seq_id`:


Plan 02 NÃO depende dos types regenerados (o teste de atomicidade usa `from('tenant_sequences')` apenas via `upsert` que aceita strings genéricas). Mas próximos plans que façam queries tipadas em `tenant_sequences` vão precisar disso.

## 5. Rollback (se algo der errado)

```sql
-- Reverte para o comportamento de 0001_init.sql (race-prone, mas funcional)
drop trigger if exists trg_opportunities_seq_id on opportunities;
drop function if exists set_opportunity_seq_id();

create or replace function set_opportunity_seq_id()
returns trigger
language plpgsql
as $$
begin
  if new.seq_id is null then
    select coalesce(max(seq_id), 0) + 1
    into new.seq_id
    from opportunities
    where tenant_id = new.tenant_id;
  end if;
  return new;
end;
$$;

create trigger trg_opportunities_seq_id
  before insert on opportunities
  for each row execute function set_opportunity_seq_id();

drop function if exists next_seq_id(uuid);
drop table if exists tenant_sequences;
```

Aplicar isto NO MESMO SQL Editor desfaz a migration completamente. Use apenas se o teste de atomicidade falhar de forma inesperada e for necessário voltar antes de investigar.
