# Phase 7.5 Plan 06: Formulário Público Hardened — Summary

**One-liner:** Defesa em camadas para `/r/[slug]` — BotID edge + Cloudflare Turnstile invisível + payload limits no RPC + log auditado com IP hashed — fechando T-07.5-D antes do deploy de produção.

## What Was Built

8 tasks executados sequencialmente em ~17min, 8 commits atômicos. O formulário público anônimo agora atravessa 4 camadas de defesa antes de chegar ao DB:

1. **Vercel BotID** (edge) classifica a sessão antes do Server Action carregar.
2. **Cloudflare Turnstile** (client challenge → server siteverify) exige token single-use.
3. **Server Action** valida ambos + loga tentativa em `public_form_submissions` (status pending → success/captcha_failed/invalid).
4. **RPC `create_public_opportunity`** enforce length/array/jsonb limits dentro da função SECURITY DEFINER (linha final de defesa contra storage exhaustion).

Mensagens de erro retornadas ao usuário são **pt-BR genéricas** (Acesso negado / Verificação anti-bot falhou / Não foi possível registrar). `error.message` real (ex: "formulario_extras excede 8KB") vai SOMENTE para `public_form_submissions.error_message` (auditoria interna).

## Decisions Made

**1. Não criar `lib/security/rate-limit.ts`** — CONTEXT.md `<deferred>` lock. Rate limit por janela fixa (5/10min IP + 50/h slug) precisa de Upstash Redis via Vercel Marketplace. Custo (~$0.20/1M req + complexidade de tuning) só se justifica com evidência de abuso. BotID+Turnstile+payload limits são suficientes para MVP.

**2. IP hashing por construção** — `hashIp()` THROW sem `IP_HASH_SALT`. Não é "preferência" — é defesa por falha-segura: sem salt, persistir IP é vetor de privacy leak via rainbow table trivial. Server Action captura e retorna pt-BR genérico.

**3. Token Turnstile single-use, RESET sempre** — `turnstileRef.current?.reset()` é chamado APÓS cada resposta do Server Action (sucesso ou erro). Cloudflare retorna `timeout-or-duplicate` em qualquer reuso. Sem reset, segundo submit do mesmo usuário falharia mesmo com token gerado novamente.

**4. Widget Turnstile renderizado SEMPRE no body do form** (não só no último step) — challenge dispara em background enquanto usuário avança pelo wizard. `size:invisible` + `appearance:interaction-only` mantém UX limpa.

**5. server-only stub em vitest.config.ts** — Vitest roda em Node puro e não tem o pacote `server-only` instalado (é dev-only do Next bundler). Alias para módulo vazio é padrão recomendado pela própria doc do Next.

**6. Cast `(supabase.rpc as unknown as AnyRpc)` em `log.ts` + `public-form.test.ts`** — `lib/database.types.ts` ainda não conhece as RPCs de 0007 (gen:types depende do schema do Cloud ser atualizado APÓS apply manual). Escape hatch idêntico ao usado em `atomicity.test.ts` para `tenant_sequences` (Plan 02). Cast pontual, removível após `npm run gen:types`.

**7. Write-only mode preservado** — Migration 0007 NÃO foi aplicada via `supabase db push`. Arquivo `0007_public_form_hardening.sql` + `07.5-06-MIGRATION-HANDOFF.md` (copy-paste-ready) entregues ao usuário para apply manual via Dashboard SQL Editor. Mesmo padrão de Plan 02.
