    **Observação importante sobre Postgres + RLS:**
    - SELECT/UPDATE/DELETE com `USING` clause **silenciosamente** filtram — não retornam erro, retornam 0 rows.
    - INSERT com `WITH CHECK` violação **retorna erro** (`new row violates row-level security policy`).
    - Por isso o spec de INSERT espera `error` não-nulo, enquanto SELECT/UPDATE/DELETE esperam `data === []`.

    **Estabilidade do teste:**
    - `singleFork: true` em vitest.config (Plan 01) evita interferência entre arquivos. Mas dentro do mesmo arquivo, `beforeAll` é fundamental — criar oportunidades uma vez, todos os specs leem.
    - `seedTestTenants` é idempotente — chamar em cada arquivo de teste é OK.

    **Não fazer:**
    - Não usar service-role para os specs RLS — sempre `authedClient` (senão RLS não testa).
    - Não asserir mensagens de erro específicas do Postgres (`'new row violates...'`) — versões diferentes podem mudar texto. Asserir `error !== null` é suficiente.
